# BOO-123 ÔÇö gh (GitHub CLI) als Voraussetzung + Golden Image + GitHub-Connect-Runbook + Branch-Protection-Fallback

## Summary
`gh` wird als verbindliche Voraussetzung verankert (HANDBUCH ┬ž3 + Anhang A + Pre-Flight + Golden Image),
der **GitHub-Connect pro VPS** als DE+EN-Runbook dokumentiert (zwei Auth-Ebenen: `gh auth` Ôëá `git auth`),
und der Branch-Protection-Schritt bekommt einen klaren manuellen Fallback ohne `gh` ÔÇö **unabhaengig**
von der SonarQube-Wahl.

## Why
Probelauf-Folgefragen (2026-06-02): `gh` fehlte in der Pflicht-Voraussetzungsliste (┬ž3) und im Golden
Image (Devcontainer-Dockerfile), obwohl `setup-branch-protection.sh` (BOO-29) es braucht. Zwei Auth-Ebenen
wurden verwechselt: `gh auth` (CLI/API) Ôëá `git auth` (`git push`); nur der SSH-Weg war dokumentiert, der
einfachere gh-als-HTTPS-Credential-Helper-Weg nirgends. GitHub-Connect wurde nur reaktiv (in der Branch-
Protection-Pruefung) gezeigt. Missverstaendnis ÔÇ×Branch-Protection braucht SonarQube" ÔÇö tatsaechlich kam
der manuelle Zwang von fehlendem `gh`.

## What
- **HANDBUCH ┬ž3** (DE+EN): `gh` in Pflicht-Software-Tabelle + Zwei-Auth-Ebenen-Callout.
- **HANDBUCH Anhang A** (DE+EN): `gh --version` in SOFTWARE; ACCOUNTS um gh-Connect + git-push-Auth-Wahl ergaenzt.
- **Devcontainer-`Dockerfile`** (Golden Image): `gh` via offizielles GitHub-apt-Repo (Keyring + sources.list).
- **HANDBUCH Anhang Y.2** (DE+EN): `gh`-apt-Repo-Block + **GitHub-Connect-Runbook** (headless `--with-token`,
  Scopes `repo`+`admin:repo_hook`; git-push-Auth SSH **oder** `gh auth setup-git`; `git remote -v`; SSHÔćĺHTTPS=BOO-118).
- **Bootstrap Pre-Flight 0.2** (DE+EN): prueft `gh` + `gh auth status` bei GitHub-Scope.

  Klarstellung **Branch-Protection Ôëá SonarQube**.
- **Bootstrap 7.6** (DE+EN): GitHub-Connect proaktiv auf der Abschluss-Checkliste.
