# BOO-149 — Branch-Protection: required_approving_review_count 1 → 0 (BUG-5)

## Summary

`setup-branch-protection.sh` setzt `required_approving_review_count: 1`. Im Solo-/Agent-Flow gibt es aber keine Fremd-Approval, und GitHub erlaubt **keine** Self-Approval — der eigene PR lässt sich nie mergen, ohne die Protection-Regel manuell zu umgehen. Fix: `required_approving_review_count` auf `0`. Die Status-Checks (CI-Gates) bleiben **Pflicht** — nur die menschliche Review-Approval, die im Solo-Flow ohnehin unerfüllbar ist, entfällt. `migrate_boo_149` wendet die Branch-Protection erneut an.

## Why

Gefunden beim CI-Post-Mortem von `bko-widerspruch-assistent` (2026-06-03) — BUG-5: der eigene PR blockiert, weil 1 Approval verlangt wird, GitHub aber Self-Approval verbietet. Im Solo-/Agent-Flow ist das ein Deadlock; die Status-Checks liefern die eigentliche Schutzwirkung. Linear: https://linear.app/owlist/issue/BOO-149

## What

- `bootstrap/references/file-templates.md` + `.en.md` (bzw. `setup-branch-protection.sh`-Block): `required_approving_review_count` von `1` auf `0`; Status-Checks bleiben unverändert Pflicht.
- `bootstrap/scripts/migrate-to-v2.sh`: neue idempotente `migrate_boo_149()` — wendet die Branch-Protection mit `required_approving_review_count: 0` erneut an; in `ALL_ISSUES` + `migrate_all()` registriert.

## Acceptance Criteria

1. `setup-branch-protection.sh` (DE+EN+Heredoc) setzt `required_approving_review_count: 0`. ✓

## Abgrenzung

Begründung leichtgewichtig (Code-Crash-Prinzip): die Schutzwirkung kommt im Solo-/Agent-Flow von den Status-Checks, nicht von einer im selben Flow unerfüllbaren Approval-Pflicht. Bei echtem Mehr-Personen-Review kann der Operator den Wert projektspezifisch wieder anheben.
