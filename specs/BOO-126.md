# BOO-126 — Design-Story-Handling klären (Discovery): Default ratifiziert

## Summary
Discovery-/Decision-Story: **Default ratifiziert** — eine Design-Story = „implement gegen eine
Design-Referenz" über die normale Pipeline (`/ideation` → `/implement`), **kein** `change_type: design`,
kein Template, keine `/ideation`-Auto-Erkennung. Festgehalten als ADR + Abgrenzungs-Absatz (DE+EN).

## Why
Probelauf-Folgefrage, bewusst „strukturiert später". Operator-Default-These (2026-06-02): Design-Story =
implement gegen Referenz. Zu entscheiden war, ob es darüber hinaus einen eigenen Story-Typ braucht.
Befund: kein design-spezifischer Story-Typ existiert; `change_type` war nur Vorschlag; die Mechanik
„Code gegen Design-Referenz" (ARCHITECTURE_DESIGN.md §9) existiert bereits.

## Entscheidung (ratifiziert)
- Design-Story läuft über die normale Pipeline; Referenz via `ARCHITECTURE_DESIGN.md §9`; `/implement`
  verifiziert gegen Referenz + a11y/Lighthouse (BOO-45).
- **Kein** `change_type: design`, kein Template, keine Auto-Erkennung — Leichtgewicht-Prinzip.
- **Begründung (Karpathy):** Code-Output hat Ground Truth (verifizierbar) → framework-tauglich;
  Design-Geschmack hat keinen Auto-Check → kein Gate → gehört nicht in die Pflicht-Pipeline.
- **Abgrenzung:** Coding-Framework *setzt Design um*; reine Gestaltung = opt-in externe Skills
  (design-md-generator, lumen-*, Pencil/Webflow-MCP).
