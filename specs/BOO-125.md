# BOO-125 — Bootstrap: projekt-spezifische MCP-Abfrage (Vercel/Apify/custom) bei Frontend + Frontend-Combo

## Summary
Neuer Block-D-Schritt **D.7**: bei `STACK = Frontend/Full-Stack` optionale Frage nach projekt-spezifischen
MCP-Servern (Vorschlag Vercel/Apify oder eigene Quelle), Registrierung via `references/mcp-setup.md` +
Hinweis auf `requires_toolsets`. Plus Vercel-Deploy-Klarstellung und Frontend-Combo-Abbildung.

## Why
Probelauf-Folgefrage: Frontend-Projekt auf Vercel → wie kommen projekt-spezifische Tools/MCPs rein? Der
Bootstrap fragte MCP nur eng (Perplexity/OpenRouter, Miro, Linear), keine generische „weitere MCP-Server?".
Verwechslungsgefahr: Vercel-Deploy braucht **keinen** MCP (GitHub-Integration), der MCP ist nur für
Agent-Interaktion (Logs/Env lesen).

## What
- `bootstrap/SKILL.md` + `.en.md`, neue **D.7**: bei STACK b/c Frage „weitere MCP-Server? Vercel/Apify oder
  eigene Quelle [ja/nein]".
- Bei ja: Registrierung gemäß `mcp-setup.md` + `tools_available` ergänzen + Hinweis, dass die nutzende
  Story/Skill den MCP in `requires_toolsets` deklariert (Bootstrap richtet nur ein).
- Vercel-Klarstellung: Deploy = Vercel↔GitHub-Integration (kein MCP); MCP nur für Agent-Interaktion.
- Frontend-Combo: „Node-Backend + Frontend" = `c) Full-Stack`; `b) Frontend` = reines Frontend.

## Acceptance Criteria
1. Block D MCP-Frage bei STACK = Frontend/Full-Stack. ✓ (D.7)
2. Registrierung gemäß `mcp-setup.md` + `requires_toolsets`-Hinweis. ✓
3. Vercel-Deploy-Klarstellung (Integration vs. MCP). ✓
4. Frontend-Combo über `c) Full-Stack` klargestellt. ✓
5. DE+EN. ✓

## Result
Umgesetzt im Branch `feat/boo-stack-mcp-branching`. Hängt an BOO-115 (Block-D-Flow). Leichtgewicht:
opt-in, Default nein, jederzeit später ergänzbar.
