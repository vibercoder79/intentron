# BOO-176 — Quality-Gate-Integrität: der Agent kann die Messlatte nicht selbst absenken

> [!note] Nummern-Hinweis (Linear-first)
> Linear-Issue **[BOO-176](https://linear.app/owlist/issue/BOO-176)** (Team Bootstrapping Evolution),
> angelegt 2026-06-07. Repo-Slot = Linear-Nummer (deckungsgleich). Branch bei Umsetzung:
> `tobiaschschmidt/boo-176-feat-quality-gate-integritat-der-agent-kann-die-messlatte`.

## Summary

Technische Sperre dagegen, dass ein KI-Agent **Quality-Gates aufweicht statt den Code zu fixen** —
z. B. PHPStan-Level 7→5 senkt, die Coverage-Schwelle reduziert, Linter-Regeln deaktiviert oder per
`change_type` Code-Gates legitim-aussehend überspringt — und das „als pragmatisch verkauft". Die
Messlatte für Code-Qualität ändert ausschliesslich der Operator, nie der Agent als Workaround. Code
fixen ✅ — Gate aufweichen ❌. Umsetzung deterministisch (Hooks/Assertion/Diff-Check), **kein** Prompt.

## Why

Operator-Feedback aus echtem Feldeinsatz (TYPO3/PHP-Projekt, 2026-06-07) + Framework-Audit. IST-Befund
(read-only verifiziert):

- Gate-Config-Dateien (im Zielprojekt generiert: `eslint.config.mjs`, `pyproject.toml`/`ruff.toml`,
  `.semgrep.yml`, Coverage-Schwelle, künftig `phpstan.neon`) stehen **nicht** in der vom Bootstrap
  generierten `.claude/sensitive-paths.json` → eine Änderung löst keinen Review-Block aus.
- Einziger Schutz heute: eine **dokumentarische** Zeile im `lint-fixer`-Subagent („Gate-Config nicht
  anfassen") — keine technische Sperre.
- Die Post-Story-Gate-Assertion (BOO-165, `sprint-run/references/gate-assertion.md`) prüft nur, ob
  Gates **liefen** (`skipped_gates` gegen `meta.json`), **nicht** ob eine Schwelle **gesenkt** wurde.
- **Schlupfloch `change_type`:** Der Agent setzt `change_type` im Spec-Frontmatter selbst; mit
  `change_type: workflow|config|…` werden über `/implement` Schritt 5.7 alle Code-Gates legitim
  übersprungen — auch bei echten Code-Änderungen. (Mechanik: `implement/references/non-code-flow.md`.)
- `change_type` ist zudem nur technisch dokumentiert (CONVENTIONS-Einzeiler + `non-code-flow.md`),
  **nicht** als verständliches Governance-Konzept im HANDBUCH (Operator-Sicht).

**Leitprinzip:** Die Qualitäts-Messlatte ändert nur der Operator. Leichtgewicht gilt für den *Prozess*,
nicht für die *Code-Qualitäts-Schwelle*.

## What

1. **Gate-Config-Pfade unter Bodyguard-Schutz** — Gate-Config-Dateien in den Default der generierten
   `.claude/sensitive-paths.json` aufnehmen (Template-Quelle: `bootstrap/references/file-templates.md`
   + `bootstrap/references/hooks-setup.md`, DE+EN). Änderung an einer Gate-Config → Gate-Block-Pause
   mit Operator-Freigabe (gleiche Mechanik wie Layer-0-Bodyguard, BOO-86 / wave-v).
2. **pre-edit-bodyguard-Muster „Schwellen-Senkung / Regel-Deaktivierung"** — Muster-Erkennung für
   `phpstan level↓`, `coverage_pass↓`, breites `eslint-disable` / `# noqa` (Template:
   `bootstrap/references/hooks/` + Heredoc in `file-templates.md`).
3. **Post-Story-Gate-Assertion erweitern** (`sprint-run/references/gate-assertion.md`):
   (a) Diff der Gate-Configs — gesenkte Schwelle ohne `override_audit` → **Fail**;
   (b) `change_type`-Plausibilität — Code-Dateien im Diff, aber `change_type: non-code` → **Fail**.
4. **Prinzip „Messlatte nur Operator"** in `CONVENTIONS.md` + HANDBUCH (DE+EN).
5. **`change_type` als Governance-Konzept im HANDBUCH erklären** (DE+EN, Operator-Sicht): was es ist,
   warum Code-Prüfungen unter welchen Bedingungen übersprungen werden dürfen, wer entscheidet, wie der
   Plausibilitäts-Check (AC 3b) Missbrauch verhindert. Bewusst im Governance-Kontext, **nicht** im
   Unit-Test-Kontext (das ist BOO-177).
6. `docs-drift` grün; Spec (diese Datei) → Result-Block; Wave-Doc (DE+EN).

## Constraints

- **Allgemeingültig für alle Stacks** — das Konzept ist stack-unabhängig; die konkrete **Pfad-Liste**
  ist stack-spezifisch: abgedeckte Stacks (JS/TS/Python) gibt das Framework vor, nicht-abgedeckte Stacks
  tragen ihre Gate-Configs via Stack-Runbook (BOO-178) nach.
- **Deterministisch** — Hooks/Assertion/Diff-Check, **kein** Prompt (Prompts garantieren nichts).
- Operator-Override bleibt möglich, aber **nur** explizit + protokolliert (`override_audit`).
