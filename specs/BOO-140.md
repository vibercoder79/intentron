# BOO-140 — ESLint-Workflow + Next.js `package.json` lint-Script (`next lint` → `eslint .`)

## Summary

Sicherstellen, dass ein frisch gebootstrapptes Next.js-Projekt seinen Lint-Gate beim ersten CI-Lauf grün bekommt. Der CI-Workflow `.github/workflows/eslint.yml` ruft bereits korrekt `npx eslint .` (seit BOO-28) — kein Fix nötig. **Residual:** Wenn das Projekt mit `create-next-app` scaffolded wurde, steht in `package.json` ein `"lint": "next lint"`. `next lint` versteht die ESLint-v9-Flat-Config (`eslint.config.mjs`) nicht und bricht lokal mit `Invalid project directory provided, no such directory: .../lint` ab. Dieses Script wird auf `"lint": "eslint ."` umgebogen.

## Why

Gefunden beim ersten CI-Lauf des Next.js-16-Projekts `bko-widerspruch-assistent` (2026-06-03). Die kaputte generierte `eslint.yml` (mit `npm run lint`) stammte aus einer **älteren Skill-Version** — das aktuelle Template (Repo-SSoT) ruft schon `npx eslint .`. Verbleibt das lokale `next lint`-Script, läuft `npm run lint` weiterhin ins Leere. Linear: https://linear.app/owlist/issue/BOO-140

## What

- `bootstrap/SKILL.md` + `.en.md`, Phase 4.4: neue Render-Regel — bei Stack b/c mit vorhandenem `package.json`-`"lint": "next lint"` das Script idempotent auf `"lint": "eslint ."` umbiegen (Operator-Bestätigung als ADR).
- `bootstrap/scripts/migrate-to-v2.sh`: neue idempotente `migrate_boo_140()` — patcht `package.json` via `jq` (nur wenn `lint == next lint`), sonst Skip; in `ALL_ISSUES` + `migrate_all()` registriert.
- `bootstrap/references/verify-setup.sh`: Regressions-Guard (Sektion 7) — FAIL wenn `package.json`-`lint` `next lint` ist bzw. `eslint.yml` `next lint`/`npm run lint` enthält.

## Acceptance Criteria

1. `eslint.yml`-Template nutzt `npx eslint .` (unverändert korrekt, verifiziert). ✓
2. SKILL.md (DE+EN) beschreibt die `package.json`-`lint`-Umbiegung für Frontend/Meta-Framework-Stacks. ✓
3. `migrate_boo_140()` idempotent (zweiter Lauf = Skip), via `--issue BOO-140` und `--all` aufrufbar. ✓
4. `verify-setup.sh` warnt/failt bei `next lint`-Resten. ✓
5. Nur Template-/Config-/Tooling-Dateien — kein Runtime-Code. ✓
