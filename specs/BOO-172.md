# BOO-172 — Runbook: Sprint unbeaufsichtigt auf der VPS per tmux fahren (DE+EN + Sketch)

## Summary
Leichtgewichtiges Runbook statt des verworfenen VPS-Daemons (Variante B): erklärt, wie man `/sprint-run` (oder jede langlaufende Claude-Arbeit) auf der VPS in `tmux` startet, sodass sie das Zuklappen des Laptops / einen SSH-Abbruch überlebt. Mit Render-Loop-Sketch (DE+EN).

## Why
Operator-Klärung (2026-06-07): `claude` läuft zwar auf der VPS, aber die Sitzung hängt an der SSH-Verbindung — Laptop zu → Agent stirbt. `tmux` entkoppelt den Prozess. Ein echter Daemon (cron/systemd) wäre Overkill; der Operator hat ihn bewusst verworfen. Bleibt: das per-tmux-Vorgehen sauber dokumentieren.

## Scope-Entscheidungen (Operator)
- **Nur die einfache tmux-Empfehlung.** Die SSH-Host-Alias-/Auto-tmux-Konfiguration wurde **bewusst rausgelassen** (zu fummelig, bricht zudem VS Code Remote-SSH).
- Als **Empfehlung** formuliert, nicht als Pflicht.

## What
- **Neu `docs/runbooks/sprint-unattended-tmux.md` (+ `.en.md`):** Problem (SSH-Tod) → Lösung (tmux) → 5 Schritte (SSH → `tmux new -A -s` → `claude` → `/sprint-run` → `Strg+B D` → `tmux attach -t`) → Cheatsheet → „wann nötig" → bewusste Grenze (kein Daemon: kein Auto-Start/Selbstheilung/Reboot-Persistenz). Klarstellung: Mac-Terminal **oder** VS-Code-Terminal gleichwertig; gilt für jede langlaufende Remote-Arbeit.
- **Sketch `sprint-unattended-tmux.{excalidraw,png}` (+ `.en.*`):** „ohne tmux (rot, stirbt) vs. mit tmux (grün, läuft weiter)". Im **Render-Loop** erstellt (excalidraw → SVG via `excalidraw_export` → PNG via `rsvg-convert` → visuelle Sichtprüfung → 1 Korrektur-Iteration) und im Runbook eingebettet.
- **Verlinkung:** `docs/runbooks/sprint-run.{md,en.md}` Verweise + `docs/onboarding/artefakt-landkarte.{md,en.md}` Runbook-Liste.
