# BOO-152 — Klon-Portabilität: settings.json + Hooks auf $CLAUDE_PROJECT_DIR + install-hooks.sh

## Summary
Macht ein gebootstrapptes Projekt **klon-portabel**, damit das Multi-User-VPS-Szenario (BOO-151) technisch sauber funktioniert. Vorher verdrahteten `settings.json` + Hook-Scripts **absolute Pfade** (`{{PROJECT_PATH}}`) — ein fremder Klon erbte die Pfade des Erstellers, Runtime-Hooks griffen ins Leere. Lösung (Weg A): Pfade auf **`$CLAUDE_PROJECT_DIR`** umstellen + ein **`scripts/install-hooks.sh`** (versioniertes `.githooks/` + `core.hooksPath`) für den nativen `pre-commit` + fertiger Onboarding-Prompt im Runbook.

## Befund (verifiziert 2026-06-04)
- `.claude/` (settings.json + Runtime-Hooks) ist committet → kommt mit dem Klon. **Gut.**
- ABER: Hook-Pfade darin absolut (`{{PROJECT_PATH}}`, `WORKSPACE="{{PROJECT_PATH}}"`) → nicht portabel über User-/Maschinen-Grenzen.
- Native `.git/hooks/pre-commit` (BOO-4) wird gar nicht geklont.

## What
- **`bootstrap/SKILL.md` (+ `.en.md`):** settings.json-Hook-Commands auf `bash "$CLAUDE_PROJECT_DIR/.claude/hooks/…"`; Phase 4.6 um nativen `pre-commit` → `.githooks/` + `install-hooks.sh` ergänzt; Portabilitäts-Hinweis. `bootstrap` 3.39.0 → 3.40.0.
- **`references/file-templates.md` (+ `.en.md`):** `WORKSPACE="{{PROJECT_PATH}}"` → `WORKSPACE="${CLAUDE_PROJECT_DIR:-$(git rev-parse --show-toplevel 2>/dev/null || pwd)}"`; neue Sektion `scripts/install-hooks.sh` (`core.hooksPath=.githooks`, idempotent).
- **`docs/runbooks/multi-user-vps.md` (+ `.en.md`):** fertiger Operator-Prompt (clone → `install-hooks.sh` → `generate-environment-json.sh` → `verify-setup.sh`).
- **HANDBUCH Anhang U + Y Pro-Projekt-Checkliste (DE+EN):** Hook-Schritt = `bash scripts/install-hooks.sh`.

## Acceptance Criteria
1. settings.json + Hook-Scripts nutzen `$CLAUDE_PROJECT_DIR` (portabel über Klon-Grenzen). ✓
2. `scripts/install-hooks.sh` (versioniertes `.githooks/` + `core.hooksPath`) als Template; Bootstrap legt es an + ruft es. ✓
3. Onboarding-Prompt im Runbook (Hook-Schritt technisch korrekt). ✓
4. HANDBUCH-Checklisten verweisen auf `install-hooks.sh` (DE+EN). ✓
5. DE+EN-Parität; docs-drift-Check grün. ✓

## Abgrenzung
Runtime-Hooks (`spec-gate`, `bodyguard`, …) brauchen `install-hooks.sh` NICHT — sie kommen via `$CLAUDE_PROJECT_DIR` + committetem `.claude/` mit dem Klon. `install-hooks.sh` aktiviert nur den nativen `pre-commit` (Quality-Gate Layer 2). Baut auf BOO-151 auf.
