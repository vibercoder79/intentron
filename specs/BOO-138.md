# BOO-138 — VPS-Standard-Projektpfad: Bootstrap erfragt PROJECTS_ROOT einmal pro Maschine + Block B liest ihn als Default

## Summary
Bootstrap erfragt **einmal pro Maschine** einen Standard-Projektpfad (`PROJECTS_ROOT`, z.B. `~/projects`) und hinterlegt ihn in der globalen Operator-`~/.claude/CLAUDE.md`. Bei jedem weiteren `/bootstrap` liest **Block B Frage 1** diesen Pfad und schlägt `<PROJECTS_ROOT>/<projektname>` als **Default** vor — manueller Override bleibt jederzeit möglich. So wird die Developer-VPS zum reibungsarmen Multi-Projekt-Host (ein Setup → jedes weitere Projekt praktisch ein Enter, alle am selben Ort).

## Why
BOO-129/130 schließen den **projekt-lokalen** Doku-Loop; die **Maschinen-Ebene** fehlte: ein neues Projekt soll ohne Reibung am Standard-Ort landen. Übertragung von Tobias' Obsidian-Vault-Prinzip („Projekte anlegen" + fester Ort) auf die VPS **ohne Obsidian**. Operator-Diskussion Tobias 2026-06-03 (Ursprung: SecondBrain-Setup → VPS).

## Befund (verifiziert 2026-06-03)
- Block B Frage 1 ist heute reine **manuelle** Pfad-Eingabe (`bootstrap/SKILL.md` Block B, „neu anlegen — wo? absoluter Pfad").
- `~/projects/<projekt>/` ist nur **Runbook-Konvention** (HANDBUCH Anhang U / VPS-Setup), wird nirgends abgefragt oder hinterlegt.
- Globale Projekt-Tabelle in `~/.claude/CLAUDE.md` existiert (Phase 7.3 / `references/global-registry-update.md` §3) — aber **kein** `PROJECTS_ROOT`-Pfad und **keine** Auslese-Logik.

Override bleibt.
