# BOO-137 — Knowledge-Onboarding-Skill: Bestands-Doku → Governance-Artefakte (Routing-Rubrik + Manifest)

## Summary

Neuer **Framework-Bundle-Skill `knowledge-onboarding`** (analog `ideation` / `implement` / `bootstrap` — Source: `intentron`-Repo, **kein** `~/.claude/skills`-Skill, **kein** `publish_skill.py`-Lifecycle). Quellen-agnostisch (GitHub-Repo / lokaler Ordner+Upload / im Chat). Routet vorhandenes Projekt-Wissen **deterministisch + wiederholbar** in die Governance-Artefakte. Läuft **post-bootstrap**. Determinismus = **Routing-Rubrik (SSoT, Tier 0/1/2/3)** + persistiertes **Manifest** (`journal/knowledge-onboarding-map.yml`).

## Why

Doku-Review-Diskussion (Tobias, 2026-06-03, ADR `02 Projekte/Code-Crash Framework/Decisions/2026-06-03 Knowledge-Onboarding-Skill — Routing-Rubrik + Manifest-Determinismus.md`):

- **BOO-117** (A.2b) liest eine Quelle nur für `PROJECT_DESC`/Stack-Hint — keine Vollständigkeit.
- **`/architecture-review`** liest Code (8 Checks), nicht menschliche Dokumente.
- **Anhang U / `framework-upgrade`** ziehen Framework-Artefakte ins Repo, **ingestieren aber nicht das Kunden-Wissenspaket**.
- Lücke: kein definierter, wiederholbarer Flow „Wissenspaket → Governance-Artefakte". Heute ad-hoc, nicht-deterministisch, fabrikations-anfällig, keine Coverage-Prüfung.

Anlassfall: `bko-widerspruch-assistent` (GAP-Analyse, Legal-Skills-Recherche, README, PLAN.md, `docs/`-Context, Design-Dateien). Determinismus-Kern: Rubrik (Signale) + persistiertes Manifest statt LLM-Whim.

## What

1. **Neuer Bundle-Skill `knowledge-onboarding/`** im Framework-Repo (Top-Level, flach):
   - `SKILL.md` + `SKILL.en.md` (Workflow: 3 Adapter → Pre-Flight → Manifest lesen → Klassifikation Tier 0/1/2/3 → Vorschlag-Tabelle → Routing-Apply → Manifest schreiben → Coverage-Check).
   - `README.md` + `README.en.md` (Was/Warum/Wie kurz).
   - `references/routing-rubric.md` + `.en.md` (**SSoT** für die Routing-Tabelle, Tier 0–11).
   - `overview.excalidraw` + `.png` + `.en.excalidraw` + `.en.png` (Render-Loop, OWLIST-Farben).

2. **3 Input-Adapter (quellen-agnostisch, gleiche Pipeline):**
   (a) GitHub-Repo (shallow clone) · (b) lokaler Ordner / Upload · (c) Chat-bereitgestellt.

3. **Routing-Rubrik als SSoT (3 Tiers + Tier 0 Ausschluss):**
   - **Tier 0 — Ausschluss:** Framework-Artefakte (`CLAUDE.md`/`AGENTS.md`/`CONVENTIONS.md`/`ARCHITECTURE_DESIGN.md`/...) + Code-Verzeichnisse (`*.{ts,tsx,js,jsx,py,go,rb,java,...}`).
   - **Tier 1 — Dateiname/Pfad** (deterministisch): `gap` → Intent; `legal/dsgvo` → Compliance; `design/style-guide` → Design; `README/PLAN` → Architektur; `context/glossar` → Vokabular; `research` → Recherche; `demo/choreographie/pitch` → Demo; `onboarding/handover` → Developer-Onboarding; `prompts/` → Prompt-Library.
   - **Tier 2 — Inhalts-Signale** (regelgebunden, Stichwort-Listen aus Rubrik).
   - **Tier 3 — mehrdeutig** (≥ 2 Kategorien oder kein Match) → **Operator fragen, nicht raten**.

4. **Routing-Manifest (Determinismus-Anker):** persistiert + committed in `journal/knowledge-onboarding-map.yml` (Schema v1: `source_path`, `source_hash`, `tier`, `category`, `match_signal`, `target_artefacts`, `action`, `pinned`, `operator_note`). Re-Scan liest das Manifest **zuerst** → bereits klassifizierte Dateien behalten ihr Routing; nur neue/geänderte werden neu klassifiziert; Operator-Korrekturen mit `pinned: true` **niemals überschreiben**.

5. **Anti-Fabrikation:** vorschlagen statt blind übernehmen; pro Item matchendes Signal + Quelldatei ausgeben; Coverage-Check am Ende; Ziel-Artefakte **referenzieren** die Quelle (Docs-as-Code, kein Volltext-Duplikat) mit standardisiertem Verweis-Block (`<!-- knowledge-onboarding · BOO-137 -->`).

6. **Timing:** Post-Bootstrap.
   - **Phase 5** Skill-Auswahl: `knowledge-onboarding` im **Standard**-Tier dabei (Installation, kein Auto-Run).
   - **Block B Bestands-Detection:** `EXISTING_INFRA`-Output bekommt Flag `bestands_doku_erkannt: true|false`.
   - **Phase 7.6 Nächste Schritte:** neuer Punkt — „Wenn Bestands-Doku/Vor-Material vorhanden → `/knowledge-onboarding` ausführen".

7. **Doku (DE+EN):**
   - **HANDBUCH-Sektion** „Knowledge-Onboarding — Bestands-Doku in Governance-Artefakte routen" (verlinkt aus §7 + Anhang U).
   - **`docs/how-we-document.md`** Sektion „Bestehendes/fremdes Repo auf Stand bringen": `/knowledge-onboarding` als **Schritt 0** voranstellen (menschliche Doku zuerst, dann `/architecture-review` für Code, dann `framework-upgrade` für Artefakt-Skelette).
