## Wave AB — orphan-check Work-Item-Ausnahme (BOO-92)

**Problem vorher:** Der optionale `orphan-check.sh` (jede neue `*.md` muss im Doku-Hub `ARCHITECTURE_DESIGN.md` §9 Referenzen registriert sein) zwang **auch** Story-Specs (`specs/<PREFIX>-<NUM>.md`) und Backlog-Records (`docs/project/backlog/record-*.md`) in den Hub — obwohl die ihre eigenen Indizes haben (`specs/` = ~25 `BOO-NN.md`). Folge: Reibung pro Story oder der Operator schaltet den Hook ab.

**Was jetzt da ist:**

- Neue, per Env ueberschreibbare Variable `ORPHAN_EXCLUDE` im orphan-check-Snippet (`hooks-setup.md` DE + `.en.md` EN). Default nimmt `specs/<PREFIX>-<NUM>.md` und `docs/project/backlog/record-*.md` aus; alle anderen `.md` werden weiter gegen den Hub geprueft.
- **Konfigurierbar statt hartverdrahtet** (Anmerkung des Felds aufgegriffen) — Projekte koennen `ORPHAN_EXCLUDE` ueberschreiben.
- Klargestellt: der Pflicht-Abschnitt heisst real **§9 Referenzen** (das Feld nannte §6).
- `migrate_boo_92()` patcht eine bereits installierte `.claude/hooks/orphan-check.sh` idempotent (Backup `.bak`); ohne installierten Hook: Skip.

**Effekt:** Stories erzeugen keinen kuenstlichen Hub-Eintrag mehr; der Hub-Schutz fuer echte Doku-Artefakte bleibt erhalten.

## Wave AC — Raw-PII-in-Logs-Guard (BOO-93)

**Problem vorher:** Klartext-PII, die versehentlich in Logs landet (`logger.info("…", original_value=user.email)`), ist ein faktisches Datenleck (DSGVO Art. 5/32) — fiel aber durch alle Schichten: der Layer-0-Bodyguard (BOO-86) deckt Secrets/eval/TLS/SQL ab, **nicht** Log-Senken; der dpo-Skill (BOO-87) liefert nur Review-Guidance. Die Ausbaustufe „Bodyguard-Warnung in PII-Pfaden" war in v0.5.0 nur als Idee vermerkt.

**Was jetzt da ist:**

- **`bootstrap/references/hooks/raw-pii-guard.py`** (kanonische Single-Source) — ein statischer **AST**-Check (python3-Stdlib `ast`, kein Regex, kein Semgrep): erkennt Log-/Audit-Senken (`log.*`, `logger.*`, `audit.*`, `logging.*`, auch `self.audit_log.*`) und meldet verbotene **Keyword-Argumente, Attribut-Lesezugriffe und Variablen-Namen** (`original_value`, `plaintext`, `raw_value` …). AST ignoriert Kommentare/Strings → kaum Fehlalarme.
- **Default = Warnung**, Hard-Block via `--strict` / `STRICT=1` (konsistent mit BOO-86 gegen Alarm-Muedigkeit).
- **Projekt-Overlay** `.claude/raw-pii-guard.local` (Basis-+-Overlay-Muster) — zusaetzliche verbotene Felder oder `sink:`-Eintraege.
- **Opt-in Scaffolding:** `migrate_boo_93()` kopiert die kanonische Quelle byte-identisch nach `.claude/hooks/`; Doku + optionaler CI-Workflow `raw-pii-guard.yml` in `hooks-setup.md` (DE+EN); dpo-Querverweis in `privacy-patterns.md`.
- Eingebauter `--self-test`.

**Designentscheid:** Pattern aus dem Feld-Beitrag PP-004 **nachgebaut, kein Code uebernommen**. Bewusst **AST** (erstes AST-Werkzeug im Repo, praeziser bei Log-Senken als Regex; Operator-Entscheid 2026-06-01) und bewusst **opt-in/warn-default** statt Pflicht-Hard-Fail — damit nichts Richtung Schwergewicht kippt. Eingehaengt in die bestehende Hook-/Migrations-Konvention (kanonische Quelle, Single-Source by construction), **nicht** als loses `ci/`-Paket.

**Effekt:** Der klassische „Rohwert versehentlich geloggt"-Fehler hat erstmals einen automatischen, optionalen Waechter — fuer regulierte Projekte auditrelevant, fuer PII-freie Projekte einfach inaktiv.

## Wave AD — HANDBUCH-Anhang Y: VPS/Cloud-Team-Runbook (BOO-94)

**Effekt:** Das Betriebswissen fuer Team-VPS ist jetzt erstklassiger, auffindbarer, zweisprachiger HANDBUCH-Bestandteil mit Diagramm — kein loses standalone Doc mehr. Reine Doku/Diagramm, kein Verhaltens-Change.

## Wave AE — raw-pii-guard ruff-clean + Hook-Lint-Gate (BOO-95)

**Problem vorher:** Die in Wave AC ausgelieferte `raw-pii-guard.py` bestand ein gaengiges striktes Ruff-Profil (`line-length 100`, flake8-bandit `S`) nicht — **9× E501** (deutsche Docstrings/f-strings/argparse-Hilfetexte) + **1× S607** (`subprocess.run(["git", …])` mit partiellem Pfad). Bei der Feld-Installation blockierte das deren Pre-Commit-Gate; das Feld musste eine `per-file-ignore`-Ausnahme bauen. Ironie: der Security-Guard scheitert am Security-Linter. Ursache: das Framework lintet seine **eigenen** ausgelieferten Python-Hooks bisher nicht.

**Was jetzt da ist:**

- **Teil A — Quelle geglaettet (kein Verhaltens-Change):** `git` via `shutil.which()` zum absoluten Pfad aufgeloest (fixt S607); verbleibendes S603 (advisory, nicht autofixbar) per gezieltem `# noqa: S603` mit Begruendung am Aufruf (feste argv, kein shell, kein User-Input). Alle 9 E501 umbrochen (mehrzeiliger Docstring, neue Helfer-Methode `_record()` fuer die `findings.append`-Zeilen, geklammerte `strict`-Zuweisung, Fortsetzungszeilen). `--self-test` weiterhin gruen.
- **Teil B — Hook-Lint-Gate (Ursachen-Fix):** `bootstrap/references/hooks/ruff.toml` (Profil `line-length 100`, `select = E,F,S`, Single-Source lokal + CI) + CI-Workflow `.github/workflows/ruff-hooks.yml` (gepinntes Ruff 0.15.11), der die kanonischen Hooks bei jedem Hook-/Profil-Touch lintet. Kuenftige `*.py`-Hooks automatisch erfasst.

**Designentscheid:** S607 **echt fixen** (absoluter Pfad ist sicherheitlich besser), S603 **gezielt unterdruecken** statt blanket-ignore — das Gegenteil der Downstream-Notloesung. Zielprofil `line-length 100` (Feld-Vorgabe), nicht 88.

**Effekt:** Der ausgelieferte Guard besteht jetzt selbst ein striktes Bandit-Profil; Downstream braucht keine Ausnahme mehr. Self-Dogfooding via CI verhindert Wiederholung. Quelle: Upstream-Feld-Feedback privacy-proxy / 2XT (Martin), 2026-06-01.

## Wave AB — orphan-check Work-Item Exception (BOO-92)

**Problem before:** The optional `orphan-check.sh` (every new `*.md` must be registered in the doc hub `ARCHITECTURE_DESIGN.md` §9 References) forced **even** story specs (`specs/<PREFIX>-<NUM>.md`) and backlog records (`docs/project/backlog/record-*.md`) into the hub — even though those have their own indexes (`specs/` = ~25 `BOO-NN.md`). Consequence: friction per story, or the operator disables the hook.

**What's there now:**

- New, env-overridable variable `ORPHAN_EXCLUDE` in the orphan-check snippet (`hooks-setup.md` DE + `.en.md` EN). The default excludes `specs/<PREFIX>-<NUM>.md` and `docs/project/backlog/record-*.md`; all other `.md` files continue to be checked against the hub.
- **Configurable instead of hard-wired** (field note taken up) — projects can override `ORPHAN_EXCLUDE`.
- Clarified: the mandatory section is really named **§9 References** (the field called it §6).
- `migrate_boo_92()` patches an already-installed `.claude/hooks/orphan-check.sh` idempotently (backup `.bak`); without an installed hook: skip.

**Effect:** Stories no longer produce an artificial hub entry; the hub protection for real doc artifacts is preserved.

## Wave AC — Raw-PII-in-Logs Guard (BOO-93)

**Problem before:** Plaintext PII that accidentally ends up in logs (`logger.info("…", original_value=user.email)`) is a factual data leak (GDPR Art. 5/32) — but fell through all layers: the Layer-0 bodyguard (BOO-86) covers secrets/eval/TLS/SQL, **not** log sinks; the dpo skill (BOO-87) only provides review guidance. The expansion stage "bodyguard warning in PII paths" was only noted as an idea in v0.5.0.

**What's there now:**

- **`bootstrap/references/hooks/raw-pii-guard.py`** (canonical single source) — a static **AST** check (python3 stdlib `ast`, no regex, no Semgrep): detects log/audit sinks (`log.*`, `logger.*`, `audit.*`, `logging.*`, also `self.audit_log.*`) and reports forbidden **keyword arguments, attribute reads, and variable names** (`original_value`, `plaintext`, `raw_value` …). AST ignores comments/strings → barely any false positives.
- **Default = warning**, hard block via `--strict` / `STRICT=1` (consistent with BOO-86 against alarm fatigue).
- **Project overlay** `.claude/raw-pii-guard.local` (base-plus-overlay pattern) — additional forbidden fields or `sink:` entries.
- **Opt-in scaffolding:** `migrate_boo_93()` copies the canonical source byte-identically into `.claude/hooks/`; docs + optional CI workflow `raw-pii-guard.yml` in `hooks-setup.md` (DE+EN); dpo cross-reference in `privacy-patterns.md`.
- Built-in `--self-test`.

**Design decision:** Pattern from field contribution PP-004 **rebuilt, no code adopted**. Deliberately **AST** (first AST tool in the repo, more precise on log sinks than regex; operator decision 2026-06-01) and deliberately **opt-in/warn-default** instead of mandatory hard-fail — so that nothing tips toward heavyweight. Hooked into the existing hook/migration convention (canonical source, single-source by construction), **not** as a loose `ci/` package.

**Effect:** The classic "raw value accidentally logged" mistake has, for the first time, an automatic, optional guardian — audit-relevant for regulated projects, simply inactive for PII-free projects.

**Problem before:** The VPS team setup knowledge existed as a standalone runbook (`docs/runbooks/vps-team-setup.md`, DE) — valuable, but not linked in the canonical HANDBUCH, not bilingual, without a diagram. The other deployment/operations topics (appendices P/Q/R/S/T/U/V), by contrast, are numbered, DE+EN-parity appendices.

**What's there now:**

- **Appendix Y "VPS/Cloud Team Runbook"** in HANDBUCH.md (DE) + HANDBUCH.en.md (EN), parity-matched, linked in the table of contents: deployment scenario & prerequisites, once-per-VPS vs. per-project (git hooks per repo!), onboarding project 2..N + brownfield, team (CODEOWNERS/branch protection), decisions (Docker vs. direct install, git-local vs. GitHub, sovereignty), four-layer on a headless VPS, quick reference.

## Wave AE — raw-pii-guard ruff-clean + Hook Lint Gate (BOO-95)

**Problem before:** The `raw-pii-guard.py` shipped in Wave AC did not pass a common strict Ruff profile (`line-length 100`, flake8-bandit `S`) — **9× E501** (German docstrings/f-strings/argparse help texts) + **1× S607** (`subprocess.run(["git", …])` with a partial path). At the field installation this blocked their pre-commit gate; the field had to build a `per-file-ignore` exception. Irony: the security guard fails the security linter. Cause: the framework had not been linting its **own** shipped Python hooks until now.

**What's there now:**

- **Part A — source smoothed (no behavior change):** `git` resolved to an absolute path via `shutil.which()` (fixes S607); remaining S603 (advisory, not autofixable) suppressed with a targeted `# noqa: S603` plus justification at the call site (fixed argv, no shell, no user input). All 9 E501 wrapped (multi-line docstring, new helper method `_record()` for the `findings.append` lines, parenthesized `strict` assignment, continuation lines). `--self-test` still green.
- **Part B — hook lint gate (root-cause fix):** `bootstrap/references/hooks/ruff.toml` (profile `line-length 100`, `select = E,F,S`, single source locally + CI) + CI workflow `.github/workflows/ruff-hooks.yml` (pinned Ruff 0.15.11) that lints the canonical hooks on every hook/profile touch. Future `*.py` hooks are captured automatically.

**Design decision:** Fix S607 **for real** (an absolute path is more secure) and **targeted-suppress** S603 instead of a blanket ignore — the opposite of the downstream stopgap. Target profile `line-length 100` (field requirement), not 88.

**Effect:** The shipped guard now passes a strict Bandit profile itself; downstream no longer needs an exception. Self-dogfooding via CI prevents recurrence. Source: upstream field feedback privacy-proxy / 2XT (Martin), 2026-06-01.
