## Wave J — Privacy by Design (BOO-69)

**Problem vorher:** Security-by-Design war voll operationalisiert (security-architect + SECURITY.md + sensitive-paths-Gate), Privacy hingegen nur ein loser Hinweis. Asymmetrie.

**Was jetzt da ist:**

- **DPO-Skill** (Data Protection Officer) mit 3 Modi: ASSESS (Rechtsgrundlage + DPIA bei Planung), REVIEW (Datenschutz-Check bei Code-Aenderung), AUDIT (Verarbeitungsverzeichnis alle N Sprints). Deckt DSGVO/GDPR (EU), BDSG (DE), nDSG (CH).
- **`PRIVACY.md`-Template** mit 8 Pflicht-Sektionen (Rechtsgrundlagen Art. 6, Verarbeitungsverzeichnis Art. 30, Loeschkonzept, Betroffenenrechte Art. 15-22, Incident-Notiz Art. 33/34, ...).
- **`personal-data-paths.json`** als HARD GATE in `/implement` Schritt 5.5b — kein Commit an personenbezogenen Pfaden ohne `privacy-ok`.
- **3 Pipeline-Hooks:** `/ideation` 0e (Privacy-Pre-Flight), `/implement` 5.5b (Gate), `/sprint-review` 7c (DPO-Audit-Trigger).
- **HANDBUCH Anhang O** als Lese-Einstieg.

**Effekt:** Operator muss kein DSGVO-Experte sein — der DPO-Skill stellt die richtigen Pruef-Fragen. Privacy ist optional aktivierbar (Bootstrap A.4), aber wenn aktiv: voll operationalisiert.

## Wave K — Deployment-Szenarien + Souveraenitaets-Stack (BOO-70 + BOO-71)

**Problem vorher:** "Wie baue ich eine Coding-Factory auf VPS auf?" und "Welche EU-Alternativen gibt es zu GitHub/Anthropic/iCloud?" waren Beratungs-Themen ohne Doku.

**Was jetzt da ist:**

- **HANDBUCH Anhang P "Deployment-Szenarien"** (DE+EN) mit Decision-Matrix + 4 Topologien (Solo-Mac / Solo-VPS / Multi-User-VPS-Coding-Factory / Team-mit-Coding-Server), je mit Setup-Schritten, Secrets-Trennung, Backup-Strategie, Tradeoffs. **Mit Topologie-Sketch** (seit Doku-Visualisierungs-Pass).
- **Bootstrap-Frage A.7** (`DEPLOYMENT_SCENARIO`, Default Solo-Mac) — eine Frage, kein Interview-Fork.
- **HANDBUCH Anhang Q "Souveraenitaets-Stack-Guide"** (DE+EN) mit EU-Alternativen-Tabelle fuer 5 Komponenten (Code-Hosting, Vault-Sync, LLM, Issue-Tracker, CI) + Migrations-Anleitungen + Decision-Matrix "Wann lohnt der Switch?". **Mit Stack-Swap-Sketch.**
- **`llm_proxy_url`-Hook** in `.claude/environment.json` (Default `null`): Operator kann einen Anonymisierungs-/Souveraenitaets-Proxy (z.B. Microsoft Presidio) davorschalten. **KEINE Engine im Framework** — nur Hook-Punkt + Audit-Spur in `meta.json.llm_routing`.

**Effekt:** Beratungs-Wissen wird selbst-bedienbar. Default-Pfad (Solo-Mac, US-Stack) bleibt frictionless; regulierte Operatoren finden Orientierung.

## Wave L — Multi-Operator-Koordination (BOO-72)

**Problem vorher:** Anhang P Szenario 4 deckte 2-5 Operatoren. Aber "20 Entwickler gleichzeitig im selben Repo, die sich Doku aus Obsidian ziehen" war undokumentiert.

**Was jetzt da ist:**

- **HANDBUCH Anhang R "Multi-Operator-Koordination"** (DE+EN) mit **3-Layer-Modell** (+ Sketch):
  - **Code-Layer** skaliert nativ via Git (Branch-Strategie: Trunk-Based / Feature-Branches / GitFlow; CODEOWNERS-Pattern).
  - **Koordinations-Layer** skaliert via Workflow-States (Team-Topologien: Pool / Squad / Hybrid).
  - **Doku-Layer** ist team-kritisch — **Obsidian Vault skaliert nicht in Teams**; Repo-Docs / Confluence / Notion / SharePoint sind die Optionen (Wahl-Matrix pro Team-Groesse).
- **Vier-Augen-Konvention** fuer `review-ok` / `privacy-ok` (nicht derselbe Operator), mit `git blame`-Audit-Spur.
- **10-Schritte-Setup-Anleitung** fuer ein 20-Personen-Team.

**Effekt:** "Skaliert das?" ist kein Vertriebs-Hindernis mehr. Reine Doku — Multi-Operator-Koordination bleibt Operator-Disziplin, kein Framework-Enforcement.

## Wave M — DPO + security-architect ins Framework-Bundle (BOO-74)

**Problem vorher (Inkonsistenz aus Wave J):** HANDBUCH Anhang O versprach Privacy-by-Design als Framework-Garantie — aber der DPO-Skill lag im Nachbar-Repo `claudecodeskills`, nicht im Framework. Gleiches galt fuer security-architect.

**Was jetzt da ist:**

- **`dpo/` und `security-architect/`** sind **vendored Bundle-Skills** im Framework-Repo (Top-Level, analog `bootstrap/`, `implement/`).
- **Bootstrap clont nur noch das Framework-Repo** (v3.29.0) — alle Skills inkl. dpo + security-architect kommen aus einer Quelle. Optionale Allzweck-Skills (research, design-md-generator, ...) via Ja/Nein-Zusatzfrage.
- **Master bleibt `claudecodeskills`** (via `publish_skill.py`), Framework-Repo ist Mirror. Sync-Konvention dokumentiert.

**Effekt:** `git clone code-crash-framework` enthaelt jetzt **alles**, was Bootstrap braucht. Das Framework ist self-contained. Solo-Use ohne Framework bleibt ueber claudecodeskills erhalten.

## Wave N — Vault-Struktur + Skill-Verortung (BOO-75 + BOO-76)

**Problem vorher:** Zwei wiederkehrende Operator-Fragen ohne klaren Ort: (1) "Wie kombiniere ich Repo-Docs im Team mit meinem persoenlichen Obsidian-Vault?" (2) "Wo installiere ich die Skills eigentlich — global, pro Projekt, oder zentral auf einer Multi-User-VPS?"

**Was jetzt da ist:**

- **BOO-75 Vault-Harvest (Anhang R Layer 3):** scharfer Grundsatz "Obsidian = Solo, nicht Enterprise" + 2-Fluss-Modell (Git bidirektional / Vault-Harvest einseitig via `git post-merge`-Hook) + Sketch. Bootstrap Block B.3 5. Option `[e]` + Inline-Hinweis (doppelte Absicherung). Config-Scaffold `vault-sync-pattern.md`. Engine-Vendoring = Phase 2 (blockiert).
- **BOO-76 Skill-Installations-Strategie (Anhang S NEU):** 3 Install-Ebenen (global pro User / pro Projekt / System-Pool) + Tradeoff Update-Last vs. Pinning + Decision-Matrix pro Deployment-Szenario + Cross-Tool (Claude Code / Codex / andere via Anhang K) + Sketch. 20-Personen-VPS-Antwort: ein System-Pool `/opt/claude/skills/`, Wartungs-Owner, NICHT pro Projekt.

**Effekt:** Beide Dauer-Fragen haben jetzt je einen bebilderten Anhang. Reine Doku, kein Feature-Code.
