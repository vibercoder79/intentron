## Wave J — Privacy by Design (BOO-69)

**Problem vorher:** Security-by-Design war voll operationalisiert (security-architect + SECURITY.md + sensitive-paths-Gate), Privacy hingegen nur ein loser Hinweis. Asymmetrie.

**Was jetzt da ist:**

- **DPO-Skill** (Data Protection Officer) mit 3 Modi: ASSESS (Rechtsgrundlage + DPIA bei Planung), REVIEW (Datenschutz-Check bei Code-Aenderung), AUDIT (Verarbeitungsverzeichnis alle N Sprints). Deckt DSGVO/GDPR (EU), BDSG (DE), nDSG (CH).
- **`PRIVACY.md`-Template** mit 8 Pflicht-Sektionen (Rechtsgrundlagen Art. 6, Verarbeitungsverzeichnis Art. 30, Loeschkonzept, Betroffenenrechte Art. 15-22, Incident-Notiz Art. 33/34, ...).
- **`personal-data-paths.json`** als HARD GATE in `/implement` Schritt 5.5b — kein Commit an personenbezogenen Pfaden ohne `privacy-ok`.
- **3 Pipeline-Hooks:** `/ideation` 0e (Privacy-Pre-Flight), `/implement` 5.5b (Gate), `/sprint-review` 7c (DPO-Audit-Trigger).
- **HANDBUCH Anhang O** als Lese-Einstieg.

**Effekt:** Operator muss kein DSGVO-Experte sein — der DPO-Skill stellt die richtigen Pruef-Fragen. Privacy ist optional aktivierbar (Bootstrap A.4), aber wenn aktiv: voll operationalisiert.

## Wave K — Deployment-Szenarien + Souveraenitaets-Stack (BOO-70 + BOO-71)

**Problem vorher:** "Wie baue ich eine Coding-Factory auf VPS auf?" und "Welche EU-Alternativen gibt es zu GitHub/Anthropic/iCloud?" waren Beratungs-Themen ohne Doku.

**Was jetzt da ist:**

- **HANDBUCH Anhang P "Deployment-Szenarien"** (DE+EN) mit Decision-Matrix + 4 Topologien (Solo-Mac / Solo-VPS / Multi-User-VPS-Coding-Factory / Team-mit-Coding-Server), je mit Setup-Schritten, Secrets-Trennung, Backup-Strategie, Tradeoffs. **Mit Topologie-Sketch** (seit Doku-Visualisierungs-Pass).
- **Bootstrap-Frage A.7** (`DEPLOYMENT_SCENARIO`, Default Solo-Mac) — eine Frage, kein Interview-Fork.
- **HANDBUCH Anhang Q "Souveraenitaets-Stack-Guide"** (DE+EN) mit EU-Alternativen-Tabelle fuer 5 Komponenten (Code-Hosting, Vault-Sync, LLM, Issue-Tracker, CI) + Migrations-Anleitungen + Decision-Matrix "Wann lohnt der Switch?". **Mit Stack-Swap-Sketch.**
- **`llm_proxy_url`-Hook** in `.claude/environment.json` (Default `null`): Operator kann einen Anonymisierungs-/Souveraenitaets-Proxy (z.B. Microsoft Presidio) davorschalten. **KEINE Engine im Framework** — nur Hook-Punkt + Audit-Spur in `meta.json.llm_routing`.

**Effekt:** Beratungs-Wissen wird selbst-bedienbar. Default-Pfad (Solo-Mac, US-Stack) bleibt frictionless; regulierte Operatoren finden Orientierung.

## Wave L — Multi-Operator-Koordination (BOO-72)

**Problem vorher:** Anhang P Szenario 4 deckte 2-5 Operatoren. Aber "20 Entwickler gleichzeitig im selben Repo, die sich Doku aus Obsidian ziehen" war undokumentiert.

**Was jetzt da ist:**

- **HANDBUCH Anhang R "Multi-Operator-Koordination"** (DE+EN) mit **3-Layer-Modell** (+ Sketch):
  - **Code-Layer** skaliert nativ via Git (Branch-Strategie: Trunk-Based / Feature-Branches / GitFlow; CODEOWNERS-Pattern).
  - **Koordinations-Layer** skaliert via Workflow-States (Team-Topologien: Pool / Squad / Hybrid).
  - **Doku-Layer** ist team-kritisch — **Obsidian Vault skaliert nicht in Teams**; Repo-Docs / Confluence / Notion / SharePoint sind die Optionen (Wahl-Matrix pro Team-Groesse).
- **Vier-Augen-Konvention** fuer `review-ok` / `privacy-ok` (nicht derselbe Operator), mit `git blame`-Audit-Spur.
- **10-Schritte-Setup-Anleitung** fuer ein 20-Personen-Team.

**Effekt:** "Skaliert das?" ist kein Vertriebs-Hindernis mehr. Reine Doku — Multi-Operator-Koordination bleibt Operator-Disziplin, kein Framework-Enforcement.

## Wave M — DPO + security-architect ins Framework-Bundle (BOO-74)

**Problem vorher (Inkonsistenz aus Wave J):** HANDBUCH Anhang O versprach Privacy-by-Design als Framework-Garantie — aber der DPO-Skill lag im Nachbar-Repo `claudecodeskills`, nicht im Framework. Gleiches galt fuer security-architect.

**Was jetzt da ist:**

- **`dpo/` und `security-architect/`** sind **vendored Bundle-Skills** im Framework-Repo (Top-Level, analog `bootstrap/`, `implement/`).
- **Bootstrap clont nur noch das Framework-Repo** (v3.29.0) — alle Skills inkl. dpo + security-architect kommen aus einer Quelle. Optionale Allzweck-Skills (research, design-md-generator, ...) via Ja/Nein-Zusatzfrage.
- **Master bleibt `claudecodeskills`** (via `publish_skill.py`), Framework-Repo ist Mirror. Sync-Konvention dokumentiert.

**Effekt:** `git clone intentron` enthaelt jetzt **alles**, was Bootstrap braucht. Das Framework ist self-contained. Solo-Use ohne Framework bleibt ueber claudecodeskills erhalten.

## Wave J — Privacy by Design (BOO-69)

**Problem before:** Security-by-Design was fully operationalized (security-architect + SECURITY.md + sensitive-paths gate), whereas privacy was only a loose hint. Asymmetry.

**What is there now:**

- **DPO skill** (Data Protection Officer) with 3 modes: ASSESS (legal basis + DPIA at planning), REVIEW (data protection check at code change), AUDIT (record of processing activities every N sprints). Covers DSGVO/GDPR (EU), BDSG (DE), nDSG (CH).
- **`PRIVACY.md` template** with 8 mandatory sections (legal bases Art. 6, record of processing activities Art. 30, deletion concept, data subject rights Art. 15-22, incident note Art. 33/34, ...).
- **`personal-data-paths.json`** as a HARD GATE in `/implement` step 5.5b — no commit on personal-data paths without `privacy-ok`.
- **3 pipeline hooks:** `/ideation` 0e (privacy pre-flight), `/implement` 5.5b (gate), `/sprint-review` 7c (DPO audit trigger).
- **HANDBUCH Appendix O** as the reading entry point.

**Effect:** The operator does not have to be a GDPR expert — the DPO skill asks the right check questions. Privacy is optionally activatable (Bootstrap A.4), but when active: fully operationalized.

## Wave K — Deployment scenarios + sovereignty stack (BOO-70 + BOO-71)

**Problem before:** "How do I build a coding factory on a VPS?" and "Which EU alternatives are there to GitHub/Anthropic/iCloud?" were consulting topics without documentation.

**What is there now:**

- **HANDBUCH Appendix P "Deployment scenarios"** (DE+EN) with a decision matrix + 4 topologies (Solo-Mac / Solo-VPS / Multi-User-VPS-Coding-Factory / Team-with-Coding-Server), each with setup steps, secrets separation, backup strategy, tradeoffs. **With topology sketch** (since the documentation visualization pass).
- **Bootstrap question A.7** (`DEPLOYMENT_SCENARIO`, default Solo-Mac) — one question, not an interview fork.
- **HANDBUCH Appendix Q "Sovereignty stack guide"** (DE+EN) with an EU alternatives table for 5 components (code hosting, vault sync, LLM, issue tracker, CI) + migration guides + decision matrix "When is the switch worth it?". **With stack-swap sketch.**
- **`llm_proxy_url` hook** in `.claude/environment.json` (default `null`): the operator can put an anonymization/sovereignty proxy (e.g. Microsoft Presidio) in front. **NO engine in the framework** — only a hook point + audit trail in `meta.json.llm_routing`.

**Effect:** Consulting knowledge becomes self-serviceable. The default path (Solo-Mac, US stack) stays frictionless; regulated operators find orientation.

## Wave L — Multi-operator coordination (BOO-72)

**Problem before:** Appendix P scenario 4 covered 2-5 operators. But "20 developers simultaneously in the same repo who pull docs from Obsidian" was undocumented.

**What is there now:**

- **HANDBUCH Appendix R "Multi-operator coordination"** (DE+EN) with a **3-layer model** (+ sketch):
  - **Code layer** scales natively via Git (branch strategy: Trunk-Based / Feature-Branches / GitFlow; CODEOWNERS pattern).
  - **Coordination layer** scales via workflow states (team topologies: Pool / Squad / Hybrid).
  - **Documentation layer** is team-critical — **the Obsidian vault does not scale in teams**; repo docs / Confluence / Notion / SharePoint are the options (choice matrix per team size).
- **Four-eyes convention** for `review-ok` / `privacy-ok` (not the same operator), with a `git blame` audit trail.
- **10-step setup guide** for a 20-person team.

**Effect:** "Does it scale?" is no longer a sales obstacle. Pure documentation — multi-operator coordination remains operator discipline, not framework enforcement.

## Wave M — DPO + security-architect into the framework bundle (BOO-74)

**Problem before (inconsistency from Wave J):** HANDBUCH Appendix O promised Privacy-by-Design as a framework guarantee — but the DPO skill lived in the neighboring repo `claudecodeskills`, not in the framework. The same applied to security-architect.

**What is there now:**

- **`dpo/` and `security-architect/`** are **vendored bundle skills** in the framework repo (top-level, analogous to `bootstrap/`, `implement/`).
- **Bootstrap now only clones the framework repo** (v3.29.0) — all skills incl. dpo + security-architect come from one source. Optional general-purpose skills (research, design-md-generator, ...) via a yes/no additional question.
- **Master remains `claudecodeskills`** (via `publish_skill.py`), the framework repo is a mirror. Sync convention documented.

**Effect:** `git clone intentron` now contains **everything** Bootstrap needs. The framework is self-contained. Solo use without the framework remains available via claudecodeskills.

## Wave N — Vault structure + skill placement (BOO-75 + BOO-76)

**Problem before:** Two recurring operator questions without a clear place: (1) "How do I combine repo docs in a team with my personal Obsidian vault?" (2) "Where do I actually install the skills — globally, per project, or centrally on a multi-user VPS?"
