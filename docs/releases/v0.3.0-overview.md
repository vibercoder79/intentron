# INTENTRON v0.3.0 — Release-Ueberblick

Stand: 2026-05-31 · Vorgaenger-Tag: `v0.2.0` (Governance-OS, Wellen J-T)

Dieses Dokument fasst zusammen, **was seit v0.2.0 dazugekommen ist** — drei Wellen (V bis X), die aus der Auswertung des Open-Source-Plugins `agentic-security` entstanden sind. Leitgedanke: **Security/Compliance frueher und belegbarer** machen, ohne das Framework schwerer zu machen. Patterns wurden nachgebaut, **kein Code uebernommen** (agentic-security ist PolyForm-lizenziert).

> Detail-Notes pro Welle: `wave-v-layer0-bodyguard.md`, `wave-w-coverage-denominator-fix.md`, `wave-x-dpo-control-catalog.md` (alle im Ordner `docs/releases/`).

## Das grosse Bild

| Welle | Story | Was sich verbessert hat | Status |
|-------|-------|--------------------------|--------|
| **V** | BOO-86 | **Layer 0** — unsichere Patterns werden abgefangen, **bevor** die KI sie schreibt (nicht erst beim Commit) | ✅ done |
| **W** | BOO-88 | **Coverage-Gate korrigiert** — zaehlt nur noch ausfuehrbare Zeilen, blockiert getestete Stories nicht mehr faelschlich | ✅ done |
| **X** | BOO-87 | **dpo-Kontrollkatalog** — Compliance Control-fuer-Control statt KI-Aufsatz, auditor-ready, reproduzierbar | ✅ done |

Roter Faden: **Leichtgewichtig ohne Security-Kompromisse** (Designprinzip 2026-05-25). Alle drei Wellen sind dependency-frei (bash + python3-Stdlib), DE+EN, mit idempotenten Migrationen fuer Bestands-Projekte. Bewusst **verworfen**: Plugin-Packaging des Bundles (nur Komfort, kein Security-Gewinn), eigene Scanner-Engine (Scanner-Wettruesten), Datenbank fuer den Kontrollkatalog (Determinismus kommt aus Git-YAML).

## Wave V — Layer-0 Edit-Bodyguard (BOO-86)

**Problem vorher:** Linter und Gates greifen erst **nach** der Implementierung (Layer 2: CLI vor Commit) bzw. in CI (Layer 3). Der Moment, in dem unsicherer Code ueberhaupt erst entsteht — die Erzeugungs-Luecke — blieb offen.

**Was jetzt da ist:**

- **`pre-edit-bodyguard.sh`** — ein Claude-Code-**PreToolUse-Hook** auf `Edit|Write`, Geschwister zu `spec-gate.sh`. Faengt Secrets, abgeschaltete TLS-Pruefung, `eval`, SQL-Konkatenation u.a. ab, **bevor** der Code auf die Platte kommt. Neuer **Layer 0** vor Layer 2/3.
- **Muster-Kataloge** `bodyguard/patterns/` — `_universal.yml` (Secrets, sprachunabhaengig) + `python/javascript/java/c-cpp.yml`, Auswahl per Datei-Endung. Jedes Muster traegt seine **Quelle** (CWE/OWASP/gitleaks) als Audit-Beleg.
- **Schichtung:** Framework-Basis + Projekt-Overlay `.claude/bodyguard.local.yml` (Kunde erweitert, ueberlebt Updates).
- **Default = Warnung**, Hard-Block opt-in via `BODYGUARD_STRICT=1` — gegen Alarm-Muedigkeit.
- HANDBUCH Anhang V, migrate-Funktion, bootstrap 3.32.0 → **3.33.0**.

**Effekt:** Security ab Erzeugung statt ab Commit — und dabei leichtgewichtig (kuratierte Muster-Menge, kein Semgrep-Lauf pro Edit; die Tiefe bleibt Layer 2/3).

## Wave W — Coverage-Nenner-Bugfix (BOO-88)

**Problem vorher:** Das Diff-Coverage-Gate zaehlte **alle** hinzugefuegten Zeilen in den Nenner — auch Kommentare und Leerzeilen. Die sind nie ausfuehrbar, koennen also nie im Zaehler landen → die Quote wurde kuenstlich gedrueckt. Beispiel: 8 getestete Code-Zeilen + 4 Kommentare + 2 Leerzeilen = `8/14 = 57 %` → faelschlich BLOCKIERT, obwohl 100 % des Codes getestet sind. Gemeldet aus einer Feld-Installation (privacy-proxy / PP-001).

**Was jetzt da ist:**

- Der Nenner zaehlt **nur ausfuehrbare Statement-Zeilen** — aus den Daten, die das Coverage-Tool ohnehin liefert (`statementMap` bei c8, `executed_lines ∪ missing_lines` bei pytest-cov). **Keine** Regex-Kommentar-Erkennung, **keine** neue Dependency.
- **Versions-Marker** `coverage-check v2`; die Migration erkennt eine alte v1 und **ersetzt** sie (Backup `.bak`) — sonst bekaemen Bestands-Installs den Fix nie.
- Fix an allen **3 Quellstellen** synchron (DE-/EN-Template + Migrations-Heredoc).

**Effekt:** Das Gate ist wieder vertrauenswuerdig — sauber kommentierter Code wird nicht mehr bestraft. Verifiziert: 8+4+2 → 100 % (vorher 57 %), 6/8 → 75 %.

## Wave X — dpo-Kontrollkatalog + deterministischer Audit-Report (BOO-87)

**Problem vorher:** Der dpo-AUDIT-Modus lieferte eine `PRIVACY.md` als Fliesstext — eine Meinung, nicht reproduzierbar, nicht an eine Control-ID gebunden, nicht auditor-exportierbar. Fuer regulierte Zielgruppen (FINMA/BaFin) zu schwach.

**Was jetzt da ist:**

- **Kontrollkataloge als YAML** im Git: `dpo/controls/gdpr.yml` (DSGVO) + `ndsg.yml` (Schweizer nDSG — **CH-Alleinstellung**, von agentic-security nicht abgedeckt). Jedes Control mit Schema `id/titel/evidenz/check_typ/check_arg/mapsTo/quelle`.
- **Deterministischer Runner** `dpo/scripts/dpo-audit.py` (python3-Stdlib, **keine DB**) erzeugt ein reproduzierbares Report-Paar `dpo/reports/<date>_audit.md` + `.json`.
- **Ehrlicher Determinismus:** mechanische Checks (Datei vorhanden? Secret im Code? TLS aus?) → **PASS/GAP**; Urteils-Checks (Zweckbindung, Verhaeltnismaessigkeit) → **REVIEW-NEEDED** (Mensch bestaetigt — kein Vortaeuschen von Voll-Automatik).
- **Projekt-Overlay** `.claude/dpo/controls/` (BYO, ueberlebt Updates). Integration in `/sprint-review` + PRIVACY.md-Querverweis + HANDBUCH Anhang O. dpo 1.1.0 → **1.2.0**. OSCAL-Export als spaetere Ausbaustufe benannt.

**Effekt:** Compliance-Evidenz wird belegbar und reproduzierbar — Control-fuer-Control statt KI-Aufsatz. Determinismus + Versionierung kommen aus dem Git-versionierten Katalog ("welche Regel galt bei welchem Commit").

# 🇬🇧 English Version

# INTENTRON v0.3.0 — Release Overview

Status: 2026-05-31 · Predecessor tag: `v0.2.0` (Governance-OS, Waves J-T)

This document summarizes **what has been added since v0.2.0** — three waves (V to X) that emerged from the evaluation of the open-source plugin `agentic-security`. Guiding idea: make **security/compliance earlier and more provable**, without making the framework heavier. Patterns were rebuilt, **no code was taken over** (agentic-security is PolyForm-licensed).

> Detail notes per wave: `wave-v-layer0-bodyguard.md`, `wave-w-coverage-denominator-fix.md`, `wave-x-dpo-control-catalog.md` (all in the `docs/releases/` folder).

## The Big Picture

| Wave | Story | What improved | Status |
|-------|-------|--------------------------|--------|
| **V** | BOO-86 | **Layer 0** — insecure patterns are caught **before** the AI writes them (not only at commit time) | ✅ done |
| **W** | BOO-88 | **Coverage gate fixed** — now only counts executable lines, no longer falsely blocks tested stories | ✅ done |
| **X** | BOO-87 | **dpo control catalog** — compliance control-by-control instead of an AI essay, auditor-ready, reproducible | ✅ done |

Common thread: **Lightweight without security compromises** (design principle 2026-05-25). All three waves are dependency-free (bash + python3 stdlib), DE+EN, with idempotent migrations for existing projects. Deliberately **discarded**: plugin packaging of the bundle (only convenience, no security gain), an own scanner engine (scanner arms race), a database for the control catalog (determinism comes from Git-YAML).

---

## Wave V — Layer-0 Edit Bodyguard (BOO-86)

**Problem before:** Linters and gates only kick in **after** implementation (Layer 2: CLI before commit) or in CI (Layer 3). The moment in which insecure code first comes into existence — the generation gap — remained open.

**What is there now:**

- **`pre-edit-bodyguard.sh`** — a Claude-Code **PreToolUse hook** on `Edit|Write`, sibling to `spec-gate.sh`. Catches secrets, disabled TLS checking, `eval`, SQL concatenation, and more, **before** the code hits the disk. New **Layer 0** ahead of Layer 2/3.
- **Pattern catalogs** `bodyguard/patterns/` — `_universal.yml` (secrets, language-independent) + `python/javascript/java/c-cpp.yml`, selected by file extension. Every pattern carries its **source** (CWE/OWASP/gitleaks) as audit evidence.
- **Layering:** framework base + project overlay `.claude/bodyguard.local.yml` (customer extends, survives updates).
- **Default = warning**, hard block opt-in via `BODYGUARD_STRICT=1` — against alert fatigue.
- HANDBUCH Annex V, migrate function, bootstrap 3.32.0 → **3.33.0**.

**Effect:** Security from generation instead of from commit — and still lightweight (curated pattern set, no Semgrep run per edit; the depth stays at Layer 2/3).

## Wave W — Coverage Denominator Bugfix (BOO-88)

**Problem before:** The diff coverage gate counted **all** added lines into the denominator — including comments and blank lines. Those are never executable, so they can never land in the numerator → the rate was artificially depressed. Example: 8 tested code lines + 4 comments + 2 blank lines = `8/14 = 57 %` → falsely BLOCKED, even though 100 % of the code is tested. Reported from a field installation (privacy-proxy / PP-001).

**What is there now:**

- The denominator counts **only executable statement lines** — from the data the coverage tool delivers anyway (`statementMap` with c8, `executed_lines ∪ missing_lines` with pytest-cov). **No** regex comment detection, **no** new dependency.
- **Version marker** `coverage-check v2`; the migration detects an old v1 and **replaces** it (backup `.bak`) — otherwise existing installs would never get the fix.
- Fix at all **3 source locations** in sync (DE/EN template + migration heredoc).

**Effect:** The gate is trustworthy again — cleanly commented code is no longer penalized. Verified: 8+4+2 → 100 % (previously 57 %), 6/8 → 75 %.

## Wave X — dpo Control Catalog + Deterministic Audit Report (BOO-87)

**Problem before:** The dpo AUDIT mode delivered a `PRIVACY.md` as free text — an opinion, not reproducible, not bound to a control ID, not auditor-exportable. Too weak for regulated target groups (FINMA/BaFin).

**What is there now:**

- **Control catalogs as YAML** in Git: `dpo/controls/gdpr.yml` (GDPR) + `ndsg.yml` (Swiss nDSG — **CH unique selling point**, not covered by agentic-security). Each control with schema `id/titel/evidenz/check_typ/check_arg/mapsTo/quelle`.
- **Deterministic runner** `dpo/scripts/dpo-audit.py` (python3 stdlib, **no DB**) generates a reproducible report pair `dpo/reports/<date>_audit.md` + `.json`.
- **Honest determinism:** mechanical checks (file present? secret in code? TLS off?) → **PASS/GAP**; judgment checks (purpose limitation, proportionality) → **REVIEW-NEEDED** (human confirms — no faking of full automation).
- **Project overlay** `.claude/dpo/controls/` (BYO, survives updates). Integration into `/sprint-review` + PRIVACY.md cross-reference + HANDBUCH Annex O. dpo 1.1.0 → **1.2.0**. OSCAL export named as a later expansion stage.

**Effect:** Compliance evidence becomes provable and reproducible — control-by-control instead of an AI essay. Determinism + versioning come from the Git-versioned catalog ("which rule applied at which commit").
