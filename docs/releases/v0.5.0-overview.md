## Wave AA ‚Äî CONTEXT.md Ubiquitous Language (BOO-91)

**Problem vorher:** Ohne festes Vokabular erfindet die KI Synonyme (`User` / `Customer` / `Betroffener` fuer dieselbe Entitaet) ‚Üí fragmentierter Code, schlechte `grep`-Barkeit, Misalignment, Token-Verschwendung. Bei regulierter Zielgruppe ist das Vokabular zusaetzlich rechtlich definiert (`Betroffener` = DSGVO Art. 4; nDSG sagt `Bearbeitung` statt `Verarbeitung`).

**Was jetzt da ist:**

- **Vorgefuellte Framework-Basis** `bootstrap/references/context-base.md` (DE+EN): eine Tabelle `kanonisch | verboten | quelle`. Compliance-Vokabular an DSGVO/nDSG gebunden, Governance-Vokabular an INTENTRON ‚Äî **jeder Begriff mit Quelle** (Audit-Beleg).
- **Projekt-Overlay** `CONTEXT.md` im Projekt-Root: beim `/bootstrap` aus der Basis geseedet plus eine `## Projekt-Domaene`-Sektion, die der Operator fuellt. Das Overlay **ueberlebt Framework-Updates** (Migration ueberschreibt es nie).
- **Default = Guidance, kein Hard-Gate:** die KI liest `CONTEXT.md` beim Schreiben und haelt sich an das kanonische Vokabular ‚Äî sie wird gefuehrt, nicht blockiert.
- Verdrahtung: Bootstrap-Phase 4.3b + `CLAUDE.md`-Verweis, `CONVENTIONS.md`-Eintrag, HANDBUCH **Anhang X**, `migrate_boo_91` (idempotent), bootstrap 3.34.0 ‚Üí 3.35.0.

**Effekt:** Begriffe sind konsistent und ‚Äî im Compliance-Teil ‚Äî an die Rechtsgrundlage gebunden. Das schaerft die Audit-Spur (nDSG-`Bearbeitung` vs. DSGVO-`Verarbeitung`) und reduziert Misalignment, ohne Reibung durch ein Hard-Gate.

**Designentscheid:** Das Pattern stammt aus Pococks `skills`-Repo ‚Äî **nachgebaut, kein Code uebernommen**. Bewusst nur die **Guidance-Schicht**; das **Enforcement** (dpo-Control ‚ÄûVokabular folgt CONTEXT.md" + Layer-0-Bodyguard-Warnung in PII-Pfaden) ist als spaetere, opt-in Ausbaustufe ausgegliedert ‚Äî damit nichts Richtung Schwergewicht kippt.
